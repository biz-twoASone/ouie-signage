## Conventions decided during this project

- **Commits per task.** Each plan task ends with its own commit. Commit messages follow conventional-commits (`feat(db):`, `feat(fn):`, `test(db):`, `chore:`). Match the exact message specified in the plan task.
- **Timestamped migrations.** `supabase/migrations/YYYYMMDDhhmmNN_<slug>.sql`. Tightly ordered; later migrations depend on earlier ones.
- **RLS is for humans only.** Device endpoints go through Edge Functions using the Supabase service-role client with explicit `WHERE tenant_id = <claim>` filtering. Do NOT add RLS policies for device roles — that approach was considered and rejected during spec refinement.
- **No placeholders, no TODOs.** If you catch yourself about to write `// TODO: implement later`, stop and either finish it or escalate to the user.
- **TDD for non-trivial Edge Functions.** Migration tasks are not TDD (the schema SQL is itself the spec); Edge Function tasks write Deno tests first, implement, run tests, commit.
- **Edge runtime does not hot-reload.** After editing any `supabase/functions/**/*.ts` file while `supabase functions serve` is already running, the container serves the old code until restarted. Run `docker restart supabase_edge_runtime_smart-tv-video-viewer` (or kill + re-run `supabase functions serve --env-file .env.local`) before re-running Deno tests — otherwise a "fix" appears not to take effect.
- **Test runner.** Run the full Deno integration suite via `deno task test` (defined in root `deno.json`). The task passes `--env-file=.env.local` so you don't need to source env manually. If you add a test that pulls env vars directly, they'll come from `.env.local` too.
- **`.env.local` values — quote them.** `.env.local` is loaded both by `supabase functions serve --env-file .env.local` (edge runtime) and by `deno test --env-file=.env.local` (test process). When editing values, wrap in double quotes: `KEY="value"`. Unquoted JWTs containing `-` characters get truncated by bash when sourced (service-role key ending in `...yH-qQwv...` was silently truncated at `yH-` in an earlier edit).
- **Reading `.env*` files is denied — don't try.** Global `~/.claude/settings.json` denies `Read(./.env)` and `Read(./.env.*)`, which covers `.env.local`, `.env.production`, and even `.env.example`. Don't attempt `Read`/`cat` on them — you'll just hit a permission wall. The project works around this with the `dotenv` npm package wired into `dashboard/next.config.ts` (`dotenv.config({ path: "../.env.local" })`), so Next.js server actions and route handlers get secrets on `process.env.X` at runtime without Claude ever seeing file contents. Edge Functions get them via `--env-file .env.local` (same idea, different loader). **Practical rules for future sessions:** (1) reference secrets in code as `process.env.KEY_NAME` (Next) or `Deno.env.get("KEY_NAME")` (Deno) — never inline values; (2) to discover which keys exist, grep the codebase for `process.env\.` / `Deno.env.get`, check `dashboard/next.config.ts` for what dotenv loads, or ask the user — do NOT try to read `.env.example`; (3) if you need the user to add a new key, tell them the exact name and where to put it (root `.env.local` for Edge/Deno, also surfaces in Next via the dotenv wiring); (4) the `dotenv` import is intentional even though Next.js has built-in `.env.local` support — it's load-path explicit (root-level file from the `dashboard/` subdir) and documents the pattern. Don't remove it.
- **PostgREST schema cache staleness.** After `supabase db reset`, PostgREST can still serve the old schema cache; `NOTIFY pgrst, 'reload schema'` alone isn't always enough. Fix: `docker restart supabase_rest_smart-tv-video-viewer`. Same pattern as the edge runtime restart.
- **Supabase Management API PATCH for auth config is NOT a true partial update.** When PATCHing `/v1/projects/<ref>/config/auth` with only some of the `smtp_*` fields (e.g. just `smtp_user` + `smtp_pass`), Supabase nulls out all the omitted `smtp_*` fields — it treats the SMTP group as one atomic block. Always send the full set (`smtp_host`, `smtp_port`, `smtp_user`, `smtp_pass`, `smtp_admin_email`, `smtp_sender_name`, `smtp_max_frequency`) in a single PATCH. Discovered the hard way on 2026-04-22.
- **Vercel deploy from monorepo:** link the project from **inside `dashboard/`** (`cd dashboard && vercel link`), NOT from repo root. Auto-detection reads the first `package.json` it sees, so linking from repo root gave Framework = "Other" with blank `buildCommand`/`installCommand`/`outputDirectory` (stored as empty strings, which are "run nothing" overrides — different from `null`, which means "use framework default"). If you inherit a broken project state, PATCH via the v9 Vercel REST API to set `buildCommand`/`installCommand`/`outputDirectory` to `null`. Auth token lives at `~/Library/Application Support/com.vercel.cli/auth.json`.
- **Default-silent offline alerting (Plan 2.2).** A device with zero entries in `screen_uptime_rules` is NEVER alerted on, even if offline > threshold. To opt in, add a rule to the device (or a group it's a member of) covering the hours it's expected to be on. Device-level rules override group-level rules entirely — if a device has its own rules, group rules are ignored for that device.
- **Android TV project location: `android-tv/` at the repo root.** Single Gradle module `:app`. Version catalog at `android-tv/gradle/libs.versions.toml` — update versions there, not inline in `build.gradle.kts`.
- **Android prerequisites (one-time, user-installed).** Android Studio 2024.2+ with SDK 35 + Platform-Tools. `ANDROID_HOME` points to the SDK (`~/Library/Android/sdk` on default installs; this machine's is `/opt/homebrew/share/android-commandlinetools`); `adb` must be on PATH. - **Supabase URL baked at build time.** `android-tv/app/build.gradle.kts` reads `SUPABASE_URL` from a Gradle property or env var, defaulting to the prod URL. Override with `./gradlew -PSUPABASE_URL=http://10.0.2.2:54321 :app:installDebug` when testing against local Supabase (`10.0.2.2` reaches the host loopback from the emulator).
- **Device token storage.** `EncryptedSharedPreferences` in file `signage_tokens.xml`. Excluded from Android auto-backup via `res/xml/backup_rules.xml`. Only the refresh token + device_id survive process death in production; the access token is re-requested on first authed call.
- **401 → refresh → retry.** `TokenAuthenticator` serializes refreshes under a mutex. If refresh itself 401s (or throws), `TokenStore` is cleared and the app falls back to Pairing on the next `AppState` emission.
- **Android stack divergence (Plan 3a).** `compileSdk`/`targetSdk` are both 35 (plan text said 34 — overridden because the Compose BOM + core-ktx pin API 35). `minSdk` stays 26. UI uses `androidx.tv.material3.*` (Button, Text, Surface), with one exception: `CircularProgressIndicator` is imported from `androidx.compose.material3.*` because TV Material3 1.0.0 doesn't ship one. Do NOT try to unify under a single `material3` namespace; leave the mix.
- **Cancellation discipline in Kotlin suspend code (Android).** Any `try { suspendingCall() } catch (Throwable)` MUST have a preceding `catch (e: CancellationException) { throw e }` — otherwise `viewModelScope` cancellation is silently swallowed. Established in commit f651b9d and enforced by code review across Plan 3a. Pattern reference: `net/TokenAuthenticator.kt`. NOTE: `isActive` inside a `suspend fun` (not inside a `CoroutineScope.launch { }` block) requires `currentCoroutineContext().isActive` — the bare `isActive` property is only available on `CoroutineScope`. Example: `MediaSyncWorker.syncAllMissing` is a suspend fn, uses `currentCoroutineContext().isActive`; `PlaybackDirector.startTicker`'s inner `while(isActive)` is inside `scope.launch`, uses bare `isActive`.
- **R2 downloads use a separate plain OkHttp client (Plan 3b).** `named("downloader")` in `AppModule.kt` is a plain client with NO auth interceptor. R2 presigned URLs carry SigV4 in the query string; attaching `Authorization: Bearer <device_jwt>` via `AuthInterceptor` makes R2 reject with 400 Bad Request. `RunningCoordinator.downloaderHttpClient` must be the plain client; never reuse the `named("authed")` client for media downloads.
- **Android cache-root selection (Plan 3b).** `RunningCoordinator.pickCacheRoot` looks at `getExternalFilesDirs(null)` (drops the first entry = emulated-internal), adds `StorageManager.getStorageVolumes()` as secondary, picks the external candidate with most free bytes ≥ 4 GB, falls back to internal with `cache_storage_info.degraded = true` otherwise. `filesystem` is reported as `"unknown"` in 3b — detecting FAT/exFAT requires hacks not worth v1. Revisit in 3c for USB-swap UUID tracking.
- **Per-process coordinator lifecycle (Plan 3b).** `RunningCoordinator` is a Koin `single` holding one `CoroutineScope` and 4 loops (ConfigPoller 60s, HeartbeatScheduler 60s, MediaSyncWorker reactive, PlaybackDirector 1Hz) + CacheStatusReporter. `MainActivity.SignageRoot`'s `LaunchedEffect(state)` calls `coordinator.start()` on `AppState.Running` and `.stop()` otherwise. `start()` is idempotent. Activity destruction cancels the scope via `onDestroy()`.
- **Media sync is eager in 3b, not window-gated.** Spec §6.2's 02:00–05:00 sync window is scheduled but not enforced by the device — the playback loop triggers downloads whenever `desired is not cached` (cache-before-switch per spec §6.3). Revisit sync-window gating once operational data shows a need.
- **`explicitNulls = false` on the shared Json (Plan 3b).** Added to `Json { ... }` in `AppModule.kt` so nullable heartbeat payload fields don't emit `"key": null` in the wire format — cleaner logs, server treats missing as default either way.
- **Edge Function deployment lag trap.** Remote deployment is NOT automatic on source edit. Plan 3b Phase 8 acceptance revealed that `devices-heartbeat` remote was Plan-1-era (wrote `last_seen_at` + `cache_storage_info` only), missing the 4 observability fields added in commit 97103bc during Plan 2. Rule: after editing any Edge Function that the device depends on, redeploy via `pnpm dlx dotenv-cli -e .env.production -- supabase functions deploy <name1> <name2> ...` before integration testing. As of 2026-04-22, remote has 10 functions: all 8 pairing/device endpoints + media-upload-url + alerts-device-offline.
- **Android emulator screencap cannot capture SurfaceFlinger overlays.** `adb exec-out screencap -p` returns black for hardware-accelerated video surfaces (ExoPlayer's `PlayerView`). Image rendering and other Compose surfaces capture fine. For visual video verification, use `screenrecord` or real hardware — Plan 3b's video playback was proven via decoder logs (`c2.goldfish.h264.decoder` initialization + `ExoPlayerImpl: Release` end-event) rather than pixels. Plan 3c extension: the Compose `PairingScreen` itself also renders black in `screencap` (pairing code recoverable from logcat instead, via the outgoing `pairing-status?code=XXXXXX` URL parameter).
- **Foreground service owns coordinator lifetime (Plan 3c — supersedes 3b).** `SignageService` is started by `MainActivity.SignageRoot`'s `LaunchedEffect(state)` on `AppState.Running` (via `ContextCompat.startForegroundService`) and by `BootReceiver` on device boot. The service's `onCreate` calls `startForeground(NOTIFICATION_ID, notif, FOREGROUND_SERVICE_TYPE_DATA_SYNC)` (with API 34 gate — 3-arg form only on UPSIDE_DOWN_CAKE+). `onStartCommand` calls `coordinator.start()` (idempotent) and returns `START_STICKY` so the OS re-launches after memory-pressure kills. `stopService` → `onDestroy` → `coordinator.stop()`. `MainActivity` no longer holds a `RunningCoordinator` reference — it only drives the service via Intent. 3b's `coordinator.stop()` in `MainActivity.onDestroy()` is gone.
- **FCM data messages route through `SyncNowBroadcast` (Plan 3c).** `SignageMessagingService.onMessageReceived` inspects `message.data["action"]` and calls `broadcast.fire()` when it's `"sync"` (spec §6.4). The coordinator's `start()` subscribes via `syncNow.events.onEach { triggerSyncNow() }.launchIn(newScope)`. `MutableSharedFlow(extraBufferCapacity=1, onBufferOverflow=DROP_OLDEST)` means bursty FCM pushes collapse to at most one in-flight + one queued sync. `SignageMessagingService` uses `org.koin.java.KoinJavaComponent.inject` (not the Android-context extension) because the service is instantiated by Android's framework before the Koin scope attaches.
- **FCM token registration via heartbeat (Plan 3c).** The device adds `fcm_token: <token>` to its `HeartbeatPayload`; `devices-heartbeat` Edge Function conditionally writes it to `devices.fcm_token` (same conditional-write pattern as the other observability fields, guarded on `typeof === "string" && length > 0`). No dedicated `devices-fcm-register` endpoint — the heartbeat path is enough, and it converges automatically on token rotation via `FcmTokenSource.update()` from `onNewToken`. First-post-boot heartbeat may carry `fcm_token: null` because `FcmTokenSource.prime()` is fire-and-forget and hasn't resolved yet; cycle 2 (60s later) has the token. This is by design.
- **Cache LRU eviction triggers inside `MediaDownloader.download()` (Plan 3c).** `CacheManager.ensureFreeSpaceFor(bytes, referencedMediaIds)` deletes oldest non-referenced files until the target is free; returns false if no eligible candidates remain, in which case `MediaDownloader` returns `Result.InsufficientSpace` and `CacheStatusReporter.failed(mediaId, "cache full...")` fires. Never evicts currently-referenced media (playlists `items.media_id` union). Safety margin 32 MB by default.
- **Preload-via-USB scan timing (Plan 3c).** `PreloadScanner.scanOnce(config)` runs at `coordinator.start()` and on every new config emission (subscribed via `configRepo.current.onEach { ... }.launchIn(newScope)` — NOT `collectLatest`; see follow-ups in plan 3c CodeQualityReviewer). Files in `<cache_root>/../preload/` are hashed (skip via `preload_index.db` SQLite cache on size+mtime match) and sha256-matched against current-config checksums. Matches are atomic-moved (`File.renameTo`) into the media cache. Unmatched files are reported in `cache_storage_info.preload.unmatched` (filename + size + sha256) for operator visibility — never auto-deleted.
- **Error bus drains per heartbeat tick (Plan 3c — persistence landed in Plan 4).** `ErrorBus.report(kind, mediaId, message)` buffers `ErrorEvent` in a bounded FIFO (default cap 32, thread-safe via `ReentrantLock`). `HeartbeatScheduler.sendOne()` calls `drain()` and inserts into the payload's `errors_since_last_heartbeat`. Events are NOT re-enqueued on heartbeat failure — single-send-best-effort matches spec §8 semantics and avoids unbounded carryover. Plan 4 superseded the Plan-3c server-only log: `devices-heartbeat` now batch-inserts into `device_error_events` with explicit tenant scoping, UUID-regex-guarded `media_id`, ISO-regex-guarded `occurred_at` (malformed falls back to server time, not whole-batch rejection), and 500-char `message` cap. Dashboard "Recent errors" card surfaces the last 10 rows per device.
- **`google-services.json` is gitignored.** Repo-root `.gitignore` + `android-tv/app/.gitignore` both exclude it. The Firebase project the file points to MUST match `FCM_PROJECT_ID` in Supabase Edge secrets — otherwise device gets a token FCM can't route to and pushes silently vaporize. Plan 3c's emulator has `project_id: signage-ouie`, matching production. Debug variant applicationId (`com.ouie.signage.debug`) is the package registered in Firebase; release build (`com.ouie.signage`, no suffix) will need a second Android app added to the same Firebase project when v1 ships.
- **`BootReceiver` can't force-launch MainActivity on Android 14 (Plan 3c).** BOOT_COMPLETED receivers get a 20s FGS tempAllowList (sufficient to start `SignageService`), but `startActivity(..., FLAG_ACTIVITY_NEW_TASK)` from the receiver is blocked by BAL (background-activity-start) restrictions (`BAL_BLOCK`, result code 102). The plan accepts this: the service runs headless, operator opens the app via Leanback launcher on first post-boot use. MIUI TV forks may differ; revisit if a customer TV keeps Home-locked.
- **Plan 3c code-review follow-ups (non-blocking).** (a) `RunningCoordinator.start()` uses `onEach { scanner.scanOnce(it) }.launchIn(newScope)` for preload scans. `collectLatest` would cancel in-flight scans on rapid config churn (matching `MediaSyncWorker`'s pattern). Low v1 impact. (b) `HeartbeatScheduler.processStartRealtime` captures time at constructor, not OS process start, so `uptime_seconds` resets on coordinator stop/start cycles. Pre-existing from 3b; service-owned coordinator in 3c makes the stop/start rare but doesn't eliminate it. `android.os.Process.getStartUptimeMillis()` would be the fix. (c) `CacheStorageInfo.preload` uses full-qualification (`com.ouie.signage.preload.PreloadStatus?`) instead of an import — inconsistent with the `ErrorEvent` import in the same file. Cosmetic.
- **FCM delivery latency is visible on the dashboard (Plan 4).** `FcmReceiptTracker` is a Koin single holding a `@Volatile var lastAt: Instant?`. `SignageMessagingService.onMessageReceived` calls `tracker.mark()` on EVERY push regardless of `action` (we care that the push arrived, not what it asked for). `HeartbeatScheduler.sendOne()` reads `tracker.current()?.toString()` into `last_fcm_received_at`. Server-side `devices-sync-now` stamps `devices.last_sync_now_dispatched_at = now()` via service-role UPDATE before firing FCM (separate from the push so DB failure doesn't block the push and vice versa). Dashboard device detail computes delta in-browser: green "Delivered in X.Xs" when received > dispatched, muted "Dispatched Xs ago, awaiting delivery" under 60s, red "Not delivered" at 60s+ (threshold = ConfigPoller.intervalMs — if push takes longer than one poll cycle, poll already covered the sync). State-only, not history — a second Sync Now overwrites the first's dispatch timestamp. Acceptable for 8 TVs.
- **Playback state in heartbeat (Plan 4).** `PlaybackDirector` implements `PlaybackStateSource` alongside `CurrentPlaylistSource`. `snapshot()` returns `PlaybackStateSnapshot(currentMediaId, stateTag)` where `stateTag ∈ {"playing", "preparing", "no_content"}`. Heartbeat ships `current_media_id` + `playback_state` fields. `devices.current_media_id` is plain `text`, NOT FK to `media` — tolerates stale references because media may be deleted between report and read. **Explicit-null gotcha**: `current_media_id: null` in the heartbeat payload MUST clear the column (Playing → Preparing transition), unlike other heartbeat fields where "missing or null" means "don't change." `devices-heartbeat` has dedicated `else if (body.current_media_id === null)` handling for this.
- **Firebase Crashlytics tagged with device_id (Plan 4).** `SignageApp.onCreate` calls `FirebaseCrashlytics.getInstance().setUserId(deviceId ?? "unpaired")` after Koin init so crashes group by device in Firebase Console. Native ANRs + uncaught Kotlin exceptions captured automatically — no `recordException` calls needed. First session post-pairing tags "unpaired" (token load is synchronous but happens pre-pair-claim); subsequent sessions use the real device_id. AndroidManifest sets `firebase_crashlytics_collection_enabled=true` because this project ships DEBUG APKs to production F&B TVs — without the meta-data, Firebase SDK defaults to collection-off for debug builds. Project auto-activates on first report receipt (no "Enable" toggle in Firebase Console UI); if the Crashlytics dashboard shows no data, force a test crash via `adb shell am crash com.ouie.signage.debug`, relaunch the app, and wait 2–5 min.
- **Plan 4 code-review follow-ups (non-blocking).** (a) `fcm_tracking.test.ts` Test 1 asserts Postgres timestamp value equals the input string (`.000Z` suffix), but Plan 1.2's fix-up verified Postgres returns `+00:00` offset notation for whole-second values. The Task 3.3 test passes, which suggests the Supabase JS client normalizes the format OR the column-specific driver path differs from the `occurred_at` timestamptz path. Worth a focused test to document the format reliably if it matters for future observability work. (b) No retention policy on `device_error_events` — bounded by device-side ErrorBus cap of 32 events/heartbeat × 1440 heartbeats/day = max ~46K rows/device/day. For 8 devices that's ~370K/day, ~135M/year at worst-case saturation. Not a near-term concern; revisit if operational data trends toward saturation.

## Execution pacing rule

User prefers **batch execution with natural-boundary pauses** over full autonomy or task-by-task approval. Group by natural phase (e.g., all migrations → pause → RLS + tests → pause → shared modules → pause → each endpoint cluster). Announce batch boundaries and report progress; continue unless the user intervenes.

## Review depth calibration

Per the subagent-driven-development skill, every task gets spec + code-quality reviews. But scale effort to complexity:

- **Trivial single-file SQL migrations with exact spec content:** one combined spec + quality review is sufficient (pattern used for Tasks 2–11).
- **Edge Function tasks (multi-file, integration logic, auth):** run spec-compliance and code-quality reviewers as separate dispatches. Different reviewers catch different classes of issue.
- **Any task touching auth, RLS, or cross-tenant isolation:** always use separate two-stage review, no shortcuts. Cross-tenant leakage is the highest-severity v1 risk.
