## Task 2: Fix single-item playlist loop (StateFlow dedup + LaunchedEffect keying)

**Root cause summary** (from session diagnosis on 2026-04-24):

For a 1-item playlist, `PlaybackDirector.advanceItem()` computes `currentIndex = (0 + 1) % 1 = 0` and emits `PlaybackState.Playing(playlistId, 0, item)` — structurally equal to the previous value because `PlaybackState.Playing` is a `data class` with `data class PlaybackItem` inside (and `File.equals()` is path-based). `MutableStateFlow` dedupes emissions via `Any.equals`, silently drops the write → Compose never recomposes → ExoPlayer stays in `STATE_ENDED`. Plan 3b acceptance missed this because that test loop had multiple items, so `index` always differed across advances.

**Fix:** Add monotonic `generation: Long` to `PlaybackState.Playing`. `advanceItem()` increments; `tick()` preserves when the item is unchanged. Propagate to `VideoPlayerHost`/`ImageSlideHost` as a `LaunchedEffect` key so same-file emits also re-prepare.

In `PlaybackDirector.kt`:

Replace the `currentIndex` declaration near line 65 with:

```kotlin
    /** 0-based index inside the currently-playing playlist. */

    /** Monotonic emission counter — see PlaybackState.Playing.generation. */
```

Replace the tail of `tick()` — from `val needsSwitch` (line 144) through the end of the function — with:

```kotlin
        val needsSwitch = (_state.value as? PlaybackState.Playing)?.playlistId != desiredPlaylistId
        if (needsSwitch) currentIndex = 0
        currentIndex = currentIndex.coerceIn(0, playlist.items.size - 1)
        val item = buildItem(playlist, cfg.media, currentIndex) ?: run {
            _state.value = PlaybackState.Preparing
            return
        }
        val prev = _state.value as? PlaybackState.Playing
        val sameItem = prev != null &&
            prev.playlistId == playlist.id &&
            prev.index == currentIndex &&
            prev.item == item
        val gen = if (sameItem) prev!!.generation else ++generationCounter
        _state.value = PlaybackState.Playing(playlist.id, currentIndex, item, gen)
    }
```

Replace `advanceItem()` with:

```kotlin
    /** Called by PlaybackScreen when the current item's duration elapsed / video ended. */
    fun advanceItem() {
        val cfg = config.value ?: return
        val s = state.value as? PlaybackState.Playing ?: return
        val pl = cfg.playlists.firstOrNull { it.id == s.playlistId } ?: return
        currentIndex = (s.index + 1) % pl.items.size
        val next = buildItem(pl, cfg.media, currentIndex) ?: return
        _state.value = PlaybackState.Playing(pl.id, currentIndex, next, ++generationCounter)
    }
```

Thread-safety note: `generationCounter` is a plain `var`. `advanceItem()` is called from the Compose main thread; `tick()` runs on the coordinator scope's dispatcher. Plain `++` on a `Long` is not atomic across threads in Kotlin/JVM. However, the existing `currentIndex` has the same property and has been live since Plan 3b without issue — 1 Hz ticks + rare advances (once per video duration / image hold) means a collision requires advance + tick to fire within the same few instructions. A lost increment here means one skipped generation, not a crash or wrong state — in the worst case the emission still differs from the previous because `++` at least writes SOME value, and `StateFlow` would still publish. Do NOT introduce `AtomicLong` here; follow pre-existing pattern. If a future Plan tightens the Director's thread-safety model, do it uniformly across `currentIndex` + `generationCounter` in one commit.

```markdown
- **Window-level FLAG_KEEP_SCREEN_ON in MainActivity (Plan 4.1).** `MainActivity.onCreate` calls `window.addFlags(FLAG_KEEP_SCREEN_ON)` before `setContent`. Defeats Android Daydream + display-sleep while the activity is foregrounded. Spec §line 17 requirement; missed in Plans 3a–c, shipped in 4.1. Does NOT override TV-firmware-level HDMI CEC inactivity power-off — customer-TV setting adjustment may be required if firmware-level sleep ever interferes (not observed on TCL Google TV).

- **PlaybackState.Playing.generation — force StateFlow emission on loop (Plan 4.1).** `MutableStateFlow` dedupes by `Any.equals`. A single-item playlist's `advanceItem()` produces a structurally-equal `Playing` state, which was silently dropped before Compose could recompose. Fix: monotonic `generation: Long` on `Playing`. `advanceItem()` always increments; `tick()` preserves when the resolved item is unchanged (so 1Hz ticks don't force ExoPlayer restarts). `VideoPlayerHost` and `ImageSlideHost` include generation as a `LaunchedEffect` key so same-file restarts re-prepare. Thread-safety of `generationCounter` intentionally mirrors the pre-existing `currentIndex` (plain `var`, not atomic) — both should be tightened together if future Plans address Director concurrency.
```
