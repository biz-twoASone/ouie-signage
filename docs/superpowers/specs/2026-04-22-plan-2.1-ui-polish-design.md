## 1. Objective

Take the dashboard from "works but basic" (Batches A–E of Plan 2) to **portfolio / client-facing-SMB-sales quality** — under a single uniform bar across all user-reachable surfaces.

The buyer persona driving this bar is an **SMB operator** (retail, F&B, clinics, salons, offices) evaluating this product against ScreenCloud / Yodeck / Raydiant incumbents. The emotional target: "this feels like a Stripe-era product," not "this feels like a 2015 CRUD app."

Done means: every one of 15 user-reachable surfaces (14 existing post-Batch-E, 1 net-new Dashboard Home) meets the uniform polish bar defined in §7 (acceptance criteria).

### Non-goals (what this pass is NOT)

- Not a rewrite. Code structure stays.
- Not a feature pass. The only net-new *feature* is Dashboard Home (§4). Everything else is visual/UX refinement.
- Not a branding system for parent ouie.app. This spec establishes Ouie Signage visuals only; other ouie.app sub-products may diverge or converge later.
- Not an accessibility overhaul beyond WCAG AA defaults.
- Not a mobile polish pass. Responsive to "not break at 375px" is required; polished-on-mobile is not.

## 2. Product identity

### Name

**Ouie Signage** — sub-function of ouie.app. Pattern: parent + function-descriptive word (cf. *Stripe Atlas*, *Notion Calendar*, *HubSpot Sales Hub*). "Signage" is industry-standard terminology buyers already search for; using it reduces cognitive-translation cost on the login page, in sales decks, and in SERPs.

### Customer-facing terminology

Database schema, API contract payloads, and TypeScript identifiers are **unchanged**. Terminology swap happens in:

1. A central `dashboard/lib/copy.ts` map for UI strings
2. URL route paths (user-facing, part of the demo-quality surface — see §2.1 below)

| Concept | DB / code (unchanged) | UI label (ships) | Route |
|---|---|---|---|
| Physical TV on the wall | `device` | **Screen** | `/screens` (was `/devices`) |
| Physical site | `store` | **Location** | `/locations` (was `/stores`) |
| Screen grouping | `device_group` | **Screen Group** | `/screen-groups` (was `/device-groups`) |
| Scheduled sequence of media | `playlist` | **Playlist** | `/playlists` (unchanged) |
| Pair-TV flow | pair-tv endpoint | **Add Screen** | `/screens/add` (was `/devices/pair`) |

Grep after polish finds **zero occurrences** of `Device`, `Store`, `Pair TV`, `Devices`, `Stores` in `dashboard/app/**/*.tsx`, `dashboard/components/**/*.tsx`, excluding types, API paths, DB field references.

### 2.1 Route-rename decision (explicit)

URL paths get renamed alongside UI labels. Rationale: a portfolio-quality product does not leak internal terminology through its URLs. A user sharing a link to `/devices/abc` would re-introduce the "Device" term we just removed from copy.

**Implementation constraint**: old routes (`/devices`, `/stores`, `/device-groups`, `/devices/pair`) must 301-redirect to new routes, not 404. Any existing in-session bookmark or pasted link must still land. Enforced in `middleware.ts`.

Files moved: `dashboard/app/devices/` → `dashboard/app/screens/`, `dashboard/app/stores/` → `dashboard/app/locations/`, `dashboard/app/device-groups/` → `dashboard/app/screen-groups/`. Server actions, route handlers, and type names inside these files remain on `device` / `store` terminology (code unchanged).

## 3. Visual system

**Direction: Modern SaaS Neutral** (Vercel/Stripe-leaning). Light-first, emerald accent, single-accent color discipline, generous whitespace, no illustrations or gradients in v1.

### 3.1 Color tokens

Defined in `dashboard/app/globals.css` using Tailwind v4 `@theme` CSS-first config.

- **Neutrals**: shadcn `neutral` scale (zinc-tinted). Three background layers only: `--bg`, `--bg-subtle`, `--bg-elevated`. No additional layers.
- **Accent**: `--color-accent` set to an OKLCH emerald (approx. `oklch(0.72 0.15 160)` light, `oklch(0.78 0.15 160)` dark). Used for primary buttons, links, focus rings, "online" status.
- **Status colors** (separate tokens, NOT reusing accent):
  - `--color-online` (emerald, but status-scoped — can diverge from accent if future design requires)
  - `--color-offline` (red-500 range)
  - `--color-warning` (amber-500 range)
  - `--color-pending` (slate-500 range)
- **Destructive**: `--color-destructive` (red-600 range)

All colors referenced via CSS variables; zero raw hex codes in component files.

### 3.2 Typography

- **Sans**: Geist Sans (Vercel's, OFL-licensed per Vercel's public statements — confirm at install time)
- **Mono**: Geist Mono — for pairing codes, IDs, byte sizes, timestamps in detail views
- **Scale**: 12 / 14 / 16 / 20 / 24 / 32 / 48 px
- **Weight**: 400 regular, 500 medium, 600 semibold. No bolder.
- **Line-height**: tight (1.2) for headings, relaxed (1.5) for body.

### 3.3 Spacing & radii

- **Spacing**: 4px baseline; major stops at 8 / 12 / 16 / 24 / 32 / 48 / 64.
- **Radii**: 6 / 8 / 12 px. No pill-shaped buttons (fights the direction).
- **Shadow**: `sm` and `md` only. No large colored glows, no "elevation-5" excess.

### 3.4 Motion

- 150ms default ease-out for state transitions.
- No bouncy / spring animations.
- `prefers-reduced-motion` fully respected — all non-essential transitions disabled.

### 3.5 Icons

Lucide (shadcn default). Consistent stroke width: 1.5. No mixing icon libraries.

### 3.6 Dark mode

Shipped in parity with light. Strategy:
- Default: follow OS via `prefers-color-scheme`.
- Manual override in user menu: Light / Dark / System.
- Persisted per-user in `localStorage` (no server round-trip).
- Accent OKLCH hue preserved; lightness flips for visual balance.
- Every surface audited in both themes during Phase 6.

### 4.2 Dashboard Home content spec

**Hero stat row** (3 `StatCard`s):
- `Screens online` — count, with "of N total" subtext
- `Screens offline` — count; clicking deep-links to filtered Screens list
- `Last sync` — aggregate timestamp (most-recent heartbeat across all screens); stale if >5min

**Needs Attention** (card, list of `ActivityItem`s):
- Offline screens (>5min without heartbeat)
- Playlists with zero items
- Media marked `pending` >10min (upload never finalized)
- Each item deep-links to its detail page

**Recent Activity** (card, list of `ActivityItem`s, last 20):
- Pairings, media uploads, playlist assignments, schedule edits

**Quick Actions** (card with 4 buttons):
- Add Screen → `/screens/add` (existing pair-TV flow)
- Upload Media → triggers media-upload flow (modal or dedicated route — implementation plan decides based on current Media library pattern)
- Create Playlist → triggers playlist-creation flow (modal or route per current Playlists list pattern)
- Create Location → triggers location-creation flow (modal or route per current Locations list pattern)

**Empty state** (zero screens tenant): hero area replaced with "Add your first screen" primary CTA + 2-line helper copy.

## 5. Design-system components

Built in `dashboard/components/ui-composed/` (distinct from shadcn primitives in `dashboard/components/ui/`). These compose shadcn primitives into project-opinionated units.

| Component | Purpose | Used by |
|---|---|---|
| `AppShell` | Sidebar + topbar + tenant switcher + user menu + theme toggle + breadcrumb slot | every page |
| `PageHeader` | Title, description, primary-action slot, secondary-actions slot | every page |
| `DataTable` | Sortable, column controls, row-hover actions, bulk-select, empty state, loading skeleton, pagination. Built on `@tanstack/react-table`. Renders semantic `<table>`. | #2, #5, #7, #8, #10, #13 |
| `StatusPill` | Online / Offline / Warning / Pending / Error — dot + label + optional timestamp | #2, #3, #15 |
| `StatCard` | Hero number + label + delta + optional sparkline slot | #15 |
| `ActivityItem` | Timestamp + actor + verb + target; deep-link target | #15 |
| `EmptyState` | Icon + headline + body + primary + secondary CTA | every list surface |
| `LoadingSkeleton` | Shimmer primitives — table rows, card, detail page | every async surface |
| `CommandPalette` | ⌘K global. Fuzzy nav across all surfaces + 4 global actions. Library: `cmdk`. | global |
| `InlineEdit` | Click-to-edit field for names (Screen, Location, Playlist, Screen Group) | #3, #6, #9, #11 |
| `ConfirmDialog` | Destructive-action confirmation (delete screen, unpair, delete media) | global |

**Unchanged shadcn primitives (reused as-is):** Button, Input, Label, Select, Checkbox, Switch, Tabs, Sheet, Dialog, Dropdown, Tooltip, Popover, Form, Toast (sonner).

### 5.1 `data-testid` contract

Every `DataTable`, `StatusPill`, `EmptyState`, primary action in `PageHeader`, and form-submit button ships with a stable `data-testid`. Naming: `{feature}-{element}[-{id}]`.

Examples:
- `screens-table`
- `screens-row-<screenId>`
- `screens-row-<screenId>-status`
- `add-screen-button`
- `pairing-code-input`
- `location-name-inline-edit`
- `playlist-composer-item-<itemId>`

**testids are stable across terminology swaps.** The UI label for `screens-table` can swap Device→Screen without the testid changing. testids serve tests; labels serve users.

## 7. Acceptance criteria

Polish pass is DONE when ALL of the following hold:

### 7.1 Visual consistency

- [ ] Every surface uses `AppShell` + `PageHeader` — no bespoke layouts
- [ ] Every list uses `DataTable` — no raw `<table>` or direct shadcn `Table` usage
- [ ] Every status indicator uses `StatusPill` — no one-off colored `Badge`
- [ ] Every list has an `EmptyState` for zero-data
- [ ] Every async surface has a `LoadingSkeleton` — no raw spinners
- [ ] Zero Tailwind arbitrary values (`w-[237px]`, `bg-[#abc]`) outside `globals.css`
- [ ] Zero raw hex codes in component files — all colors reference CSS variables
- [ ] All UI copy routes through `lib/copy.ts` — grep finds zero `Device` / `Store` / `Pair TV` in JSX

### 7.2 Dark mode

- [ ] Every surface audited in both light and dark during Phase 6
- [ ] Manual toggle + system-auto option in user menu
- [ ] All `StatusPill` variants legible in both themes (contrast ≥ WCAG AA)

### 7.3 Keyboard & density

- [ ] `CommandPalette` (⌘K) navigates to every surface and executes the 4 global actions
- [ ] `InlineEdit` works for Screens, Locations, Playlists, Screen Groups
- [ ] Primary actions have keyboard shortcuts visible in tooltips

### 7.4 Dashboard Home

- [ ] Loads in <1s on localhost against seeded DB
- [ ] Hero stats reflect real data
- [ ] Needs-Attention items deep-link correctly
- [ ] Empty state (zero-screens tenant) shows "Add your first screen"

### 7.5 Regression guard (Phase 0 smoke suite)

- [ ] `@playwright/test` installed in `dashboard/`
- [ ] `pnpm test:e2e` runs the smoke suite locally against seeded dev DB
- [ ] Smoke suite covers: magic-link login, tenant bootstrap, Location CRUD, Add Screen flow, media upload + delete, playlist assign, Sync Now returns 202
- [ ] Smoke suite passes after every phase
- [ ] Any phase-end failure blocks the next phase until fixed

### 7.6 Other regressions

- [ ] No new TypeScript errors
- [ ] No new ESLint warnings
- [ ] Lighthouse on Dashboard Home ≥ 95 (performance / accessibility / best-practices)
- [ ] No cross-tenant data leakage introduced (manual audit of any server-action refactored during polish)

## 10. Risks

Carried from brainstorming with honest framing:

1. **No prior committed regression suite.** Phase 0 creates one; prior to Phase 0, any polish-introduced regression is caught only manually. Mitigation: Phase 0 is the first phase, non-negotiable.
2. **Terminology-swap miss.** Easy to miss a hardcoded `Device` or `Store` string during refactor. Mitigation: central `lib/copy.ts`; grep-based verification in §7.1.
3. **Server-action refactor riding along with polish.** Component refactors can nudge cleanup of server actions where auth/tenant filters live. Highest-severity regression class (cross-tenant leakage). Mitigation: polish pass forbids server-action behavior changes — only presentational changes. If a server-action change is needed, it gets a dedicated commit with a dedicated review.
4. **DataTable swap-in.** Replacing raw tables changes DOM structure, pagination, URL-state interaction. Mitigation: render semantic `<table>` via tanstack; smoke suite catches basic row-count regressions.
5. **AppShell wrap.** Every page's layout changes; easy to accidentally move an auth guard. Mitigation: smoke suite login test catches broken auth immediately.
6. **Dark-mode contrast.** Status pills may become unreadable in dark. Mitigation: Phase 6 is dedicated to this; acceptance criterion 7.2 blocks merge.
7. **10-cycle budget runs mid-phase pause.** If external work (customer arrives, Android Plan 3 starts) interrupts, pause at phase boundaries, never mid-phase.
8. **`@tanstack/react-table` dependency.** ~14kb gz; widely trusted. Accepted.
9. **Over-engineering CommandPalette.** Keep it tight: fuzzy over static route list + 4 global actions. No AI, no cross-data search in v1.
10. **Route rename breaking in-session bookmarks.** Moving `/devices` → `/screens` et al. 404s any bookmark, any pasted-link to the old path. Mitigation: `middleware.ts` issues 301 redirects from old paths to new; smoke suite includes one test for each redirect.
